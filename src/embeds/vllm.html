LLM Serving · Inference Engines

vLLM

You're about to put an open-source model behind a company API for everyday questions. vLLM is the piece that turns raw model weights into a service that stays fast when the whole team hits it at once. Here's what it actually does, why it became the default, where it isn't the right tool, and how to stand one up.

Full mode — deep-dive blocks visible

The one idea

A trained LLM is just a pile of weights. Getting one token out of it is a big matrix multiply; getting many users' tokens out of it efficiently is a memory-and-scheduling problem. vLLM is a specialised server that solves exactly that problem, so a single GPU serves far more concurrent traffic than a naive model.generate() loop ever could.

Key insight

vLLM keeps the GPU as busy as physically possible by treating the KV cache like operating-system virtual memory (PagedAttention) and by admitting and retiring requests one token-step at a time (continuous batching). Those two ideas are the whole reason it's fast. Everything else — the OpenAI-compatible API, quantization, tensor parallelism — is convenience built on top.

It is an inference engine, not a framework. It sits below your application: your app does retrieval, prompt construction, and orchestration; vLLM just turns "here are some prompts" into "here are some tokens" as fast and as cheaply as the hardware allows.

What vLLM actually is

vLLM came out of UC Berkeley's Sky Computing Lab; the core idea was published as Efficient Memory Management for Large Language Model Serving with PagedAttention (SOSP 2023). It is now one of the most widely adopted open-source inference engines, with a large community and vendor backing. You use it in one of two shapes:

  • Offline batched inference — a Python LLM object you call .generate() on. Good for evals, bulk scoring, data pipelines.
  • An online server — vllm serve <model> exposes an OpenAI-compatible HTTP API (/v1/chat/completions, /v1/completions, /v1/embeddings). This is what you want for a company Q&A endpoint: any code that already talks to the OpenAI SDK works by changing one URL.

Where it sits in your stack: your app / RAG layer → vLLM (OpenAI-compatible) → GPU. vLLM owns the last hop and nothing above it.

Deep dive — the engine, briefly

Under the hood vLLM has a scheduler (decides which requests run each step and how KV memory is allocated), a KV cache manager (the paging system), and a model runner (the actual GPU forward pass, using fused attention kernels such as FlashAttention). The 2025 rewrite — the V1 engine — split the client-facing API loop and the GPU execution loop into separate processes talking over ZMQ, added near-zero-overhead prefix caching, and made the throughput-oriented defaults the standard path. As of the 0.11 release the legacy "V0" engine was removed entirely, so today "vLLM" means "V1".

Why it's fast, part 1: PagedAttention

Every token a model generates has to attend to every previous token, so the model caches the per-token key/value tensors — the KV cache. The KV cache, not the weights, is what limits how many requests you can run at once. On an 8B model in fp16 the cache costs roughly 128 KB per token; a single 8k-token conversation needs about 1 GB of it.

Naive servers pre-allocate one contiguous block per sequence sized to the maximum length it might reach. Almost every request finishes far short of that maximum, so most of the reserved memory sits empty — classic internal fragmentation. Measured KV-cache waste in those systems runs 60–80%.

The trick

PagedAttention chops the KV cache into fixed-size blocks (like OS memory pages) and hands them out on demand, non-contiguously, via a per-sequence block table. A sequence only ever holds blocks for tokens it has actually produced. Waste drops to under 4% — so the same GPU holds many more sequences, which means bigger batches, which means more throughput.

Deep dive — the block table, and a free bonus

The block table maps a sequence's logical token positions to physical KV blocks. A custom attention kernel gathers the scattered blocks at each step, so the model never sees the fragmentation. Because blocks are the unit of sharing, two sequences with an identical prefix (a shared system prompt, or the same retrieved document) can point at the same physical blocks — that's prefix caching, and it's why a fixed system prompt is nearly free after the first request. The same block machinery also makes parallel sampling and beam search cheap: forked sequences share the common blocks with copy-on-write.

PagedAttention makes vLLM faster primarily because it…

  • makes the attention matrix multiply run on faster CUDA cores
  • slashes wasted KV-cache memory, so more sequences fit at once and batches get bigger
  • compresses the model weights so the model loads faster
Throughput here is a memory story, not a matmul story. PagedAttention doesn't touch the arithmetic of attention — it removes the 60–80% of KV memory that naive contiguous allocation wastes. More free KV memory ⇒ more concurrent sequences ⇒ larger batches per GPU step ⇒ higher throughput. Weights are untouched.

Why it's fast, part 2: continuous batching

GPUs love big batches, but requests don't arrive in tidy batches and they don't finish together — one user wants a 3-token yes/no, another wants a 400-token explanation. Static batching groups requests, runs the whole group until the slowest one finishes, and only then starts the next group. Every short request in that group leaves its GPU lane idle for the rest of the group. Head-of-line blocking wastes huge amounts of the GPU.

Continuous batching (a.k.a. iteration-level scheduling) reschedules at every single decoding step: the instant a sequence emits its stop token and frees its lane, a waiting request drops straight in. No lane waits for the batch; the GPU stays saturated.

Deep dive — where it comes from, and the compounding win

Iteration-level scheduling was introduced by Orca (OSDI 2022); vLLM combines it with PagedAttention so that admitting a new request mid-flight is just "allocate a block," not "find a contiguous megabyte." The demo above uses a fixed set of 12 requests, so the win looks modest. In production it compounds: new requests keep arriving, and continuous batching folds each one into the running batch within a step, while paged memory means the batch can grow large. That interaction is where the headline "2–4× throughput, 85–92% GPU utilisation" comes from. vLLM also does chunked prefill — slicing a long prompt's prefill into pieces so it interleaves with other requests' decode steps instead of stalling them.

Compared with static batching, continuous batching changes mainly the…

  • number of GPUs each request runs on
  • precision of the model weights (fp16 → fp8)
  • granularity of scheduling — requests join and leave the running batch at each token step, not each batch
Continuous batching is a scheduling change, nothing to do with GPU count or numeric precision. Instead of "assemble a batch, run to completion, repeat," the scheduler re-forms the batch every decoding iteration, so a finished request's lane is reused immediately and a newly arrived request doesn't wait for the current group to end. That's what keeps utilisation high under bursty, mixed-length traffic.

Why the industry standardised on it

2–4×throughput vs a naive serving loop on the same GPU
<4%KV-cache waste (vs 60–80% naive)
1 URLto swap in — OpenAI-compatible API
~allpopular open models supported, fast

Beyond the two core tricks, the reasons vLLM became the default choice for self-hosting are practical:

  • OpenAI-compatible drop-in. Point your existing OpenAI client's base_url at vLLM and keep every line of app code. Migrating off a hosted API becomes a config change.
  • Model coverage, fast. New open models (Llama, Qwen, Mistral, DeepSeek, Gemma, Phi, …) usually get support within days. You're not locked to one family.
  • Scales down and up. One command on one GPU; or tensor / pipeline / data / expert parallelism across many GPUs and nodes for large models.
  • The features you actually need in prod — quantization (AWQ, GPTQ, FP8, INT8), automatic prefix caching, speculative decoding, structured/JSON output, multi-LoRA serving, streaming, and Prometheus metrics.
  • Open source, no per-token bill. You own the box and the model; cost is hardware, not API metering.

The real thing

      
These are the commands you'll actually run on the GPU box.


      
Install (GPU host, Python 3.10–3.12)

      
# uv is the fast path the docs now recommend; brings its own CUDA wheels
uv venv && source .venv/bin/activate
uv pip install vllm
# or classic: pip install vllm


      
Serve an OpenAI-compatible API

      
vllm serve meta-llama/Llama-3.1-8B-Instruct \
  --port 8000 \
  --api-key sk-company-secret \
  --gpu-memory-utilization 0.90 \
  --max-model-len 8192
# exposes /v1/chat/completions, /v1/completions, /v1/models on :8000


      
Bigger model on the same box — quantize + split across GPUs

      
vllm serve Qwen/Qwen2.5-32B-Instruct-AWQ \
  --quantization awq \
  --tensor-parallel-size 2        # shard the model over 2 GPUs


      
Call it with the stock OpenAI client — the drop-in win

      
from openai import OpenAI
client = OpenAI(base_url="http://localhost:8000/v1", api_key="sk-company-secret")

resp = client.chat.completions.create(
    model="meta-llama/Llama-3.1-8B-Instruct",
    messages=[{"role": "user", "content": "How do I reset my VPN?"}],
)
print(resp.choices[0].message.content)


      
…or straight curl

      
curl http://localhost:8000/v1/chat/completions \
  -H "Authorization: Bearer sk-company-secret" \
  -H "Content-Type: application/json" \
  -d '{"model":"meta-llama/Llama-3.1-8B-Instruct",
       "messages":[{"role":"user","content":"hello"}]}'


      
Offline batched inference (evals, bulk scoring)

      
from vllm import LLM, SamplingParams

llm = LLM(model="meta-llama/Llama-3.1-8B-Instruct")   # pulls weights from HF
params = SamplingParams(temperature=0.7, max_tokens=256)
out = llm.generate(["Explain vLLM in one sentence."], params)
print(out[0].outputs[0].text)

    


    

      
Pros, cons, and when to pick something else

      
Strengths

      
    
        
  • Best-in-class throughput per GPU under real, concurrent, mixed-length load.
    • 
        
  • Memory efficiency from PagedAttention → more users per card.
    • 
        
  • OpenAI-compatible — trivial to adopt, trivial to swap providers.
    • 
        
  • Broadest, fastest model support; rich production features.
    • 
      

      
Costs / limits

      
    
        
  • GPU-centric. It targets NVIDIA (AMD/others exist but are less trodden).
          It is not the tool for a laptop, a CPU-only box, or a single casual chat.
    • 
        
  • Tuned for aggregate throughput, not single-request latency. For the
          absolute lowest latency on one request, TensorRT-LLM can beat it.
    • 
        
  • Operational surface. You own the GPU, the weights, the memory tuning,
          and the upgrades — more knobs than a one-command tool.
    • 
        
  • Fast-moving. Flags and defaults shift between releases (the V0→V1
          migration is the big example). Pin a version in prod.

A multi-user internal Q&A API over an open model is squarely vLLM's home turf — make
          it the default. Use Ollama only for poking at models on your own laptop.
          If the workload later becomes RAG-heavy where every request shares the same large context,

Where people go wrong

      
Pitfall · OOM at startup
        
vLLM budgets KV cache from --max-model-len up front. Setting it to the
          model's full context (say 128k) when your prompts are ~1k reserves memory you'll
          never use and can OOM or crush concurrency. Set --max-model-len to your real
          ceiling, and tune --gpu-memory-utilization (0.90 is a sane start).

      
Pitfall · base vs instruct
        
Serving a base checkpoint and getting rambling, non-answering output? Use the
          -Instruct/-Chat variant and the /v1/chat/completions
          endpoint so vLLM applies the model's chat template.

      
Pitfall · benchmarking one request
        
Timing a single prompt and concluding "it's not that fast" measures the wrong thing.
          vLLM optimises aggregate throughput. Load-test with concurrency that matches
          production before you judge it — or compare it against TensorRT-LLM if single-request
          latency is truly your KPI.

      
Pitfall · quantization isn't free
        
AWQ/GPTQ/FP8 cut VRAM and can speed decode, letting you fit a bigger/faster model — but
          they can shift output quality. Validate on your own eval set, don't assume parity.

      
Pitfall · "vLLM is my RAG stack"
        
It isn't. vLLM is only the inference server. Retrieval, chunking, prompt assembly, and
          orchestration live in your application above it. (Keeping that boundary clean is
          exactly why swapping the model or the engine later stays cheap.)

    


    

      
Cheat-sheet

      
    
        
  • What: high-throughput LLM inference server. Two ideas —
          PagedAttention (paged KV cache, <4% waste) + continuous batching
          (per-step scheduling). OpenAI-compatible.
    • 
        
  • Stand up: pip install vllm →
          vllm serve <hf-model> --port 8000 --api-key ... → point your OpenAI
          client at http://host:8000/v1.
    • 
        
  • Key flags: --tensor-parallel-size N ·
          --gpu-memory-utilization 0.9 · --max-model-len ·
          --quantization awq|gptq|fp8 · --max-num-seqs ·
          --api-key · prefix caching is on by default.
    • 
        
  • Sizing rule: free_KV = gpu_mem·util − weights;
          concurrency ≈ free_KV / (kv_per_token · avg_len);
          kv_per_token = 2·layers·kv_heads·head_dim·dtype_bytes.
    • 
        
  • Reach elsewhere: laptop/CPU/single-user → Ollama; shared-prefix
          RAG/agents → try SGLang; last-drop latency on NVIDIA → TensorRT-LLM.
    • 
        
  • Numbers: ~2–4× throughput vs naive; keeps a GPU ~85–92% busy under load.
