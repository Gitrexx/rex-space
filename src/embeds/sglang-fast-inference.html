LLM Serving · Inference Infrastructure

SGLang

You're about to self-host an open model so the company gets fast, cheap answers to everyday questions. SGLang is one of the two engines you'll seriously weigh (the other is vLLM). This module is the engine itself — what it is, why the big labs run it, its honest trade-offs, and how to stand up a server. Your prompts will share a big fixed system prompt across thousands of small questions — which happens to be the exact workload SGLang was built to make cheap.

The one idea

SGLang is two things in one package: a high-performance runtime that serves an LLM behind an OpenAI-compatible HTTP API, and a small programming language for scripting multi-step LLM calls. The runtime is what you'll use day one. Its signature trick is to treat the KV cache as a reusable tree instead of throwing it away after every request.

Key insight

Every generation builds a KV cache for the prompt tokens. Most serving stacks discard it when the request ends. SGLang keeps recent caches in a radix tree keyed by token prefix, so a new request that starts with an already-seen prefix (your shared system prompt, few-shot block, or chat history) skips recomputing it. For a helpdesk bot where 10,000 questions all sit behind the same 800-token system prompt, that prefix is computed roughly once, not 10,000 times.

Hold that picture — "reuse the shared prompt prefix" — and the rest of SGLang (its benchmarks, its pros and cons, when to pick it over vLLM) falls out of it.

What SGLang actually is

The name is a portmanteau of Structured Generation Language. It started as a research project from the LMSYS / SGL team (the group behind Chatbot Arena and Vicuna) and is now a widely used open-source serving stack (sgl-project/sglang). Two layers:

  • The runtime (SRT). A GPU inference server. You point it at a Hugging Face model, it loads the weights and exposes /v1/chat/completions and friends — a drop-in OpenAI-compatible endpoint. This is the part that competes with vLLM, TGI, and TensorRT-LLM. It bundles continuous batching, tensor/data parallelism, quantization, CUDA graphs, fast attention kernels (FlashInfer), and RadixAttention prefix caching.
  • The frontend language. A Python DSL (sgl.function, sgl.gen, sgl.select, fork) for expressing programs that interleave generation, control flow, constrained decoding, and parallel branches — and it co-designs with the runtime so those branches automatically share cached prefixes. You can ignore this layer entirely and just use the OpenAI API; it's there when your call patterns get fancy.

For your use case — a server answering easy questions — you live almost entirely in the runtime and talk to it with the standard OpenAI SDK. No lock-in: swap the base URL and your existing client code works.

Deep dive — the runtime's moving parts

Under the hood the server is a set of cooperating processes: a tokenizer process, one or more scheduler + model-worker processes (one per tensor-parallel rank), and a detokenizer. The scheduler owns the batch: it decides which requests run each GPU step, manages the KV-cache memory pool, and maintains the radix tree. On top sits an HTTP server (FastAPI-style) translating OpenAI requests into the internal /generate protocol. Attention is executed by FlashInfer / Triton kernels; the model definitions live in SGLang itself, which is why support for a brand-new architecture sometimes lands here before it lands elsewhere.

The core trick: RadixAttention

A radix tree (a compressed prefix tree / trie) is the data structure at the heart of SGLang. Each edge holds a run of tokens; each path from the root spells out a prompt prefix; each node holds the KV-cache blocks for that prefix. When a request arrives, the scheduler walks the tree matching the request's tokens as far as it can. The matched portion is a cache hit — its KV cache is reused for free. Only the unmatched suffix is actually computed, and its new KV blocks are inserted as fresh nodes.

Watch it happen. Five requests arrive; the first four share a system prompt, three of them also share a few-shot block, and request #5 is a repeat. Step through and watch green (reused) grow while orange (freshly computed) shrinks.

■ reused from cache (a hit — no compute)   ■ freshly computed this request. The radix tree branches at system when request #4 skips the few-shot block, and request #5 is a full hit. Without prefix reuse, every request would recompute its whole prompt.

Deep dive — eviction and cache-aware scheduling

GPU memory is finite, so the tree can't grow forever. SGLang evicts with an LRU policy on leaf nodes: the least-recently-used leaf's KV blocks are freed first, and a node is never evicted while a running request still references it (reference counting). Two subtleties make this more than a passive cache:

  • Cache-aware scheduling. The scheduler can order and group requests to maximize prefix sharing — batching together requests that hit the same tree nodes so the shared prefix stays hot and isn't evicted between them.
  • It composes with paging. RadixAttention decides what to reuse; a paged KV memory pool decides where blocks physically live. They're orthogonal — RadixAttention is the reuse policy layered on top of block-based KV storage.

The payoff is largest exactly when prefixes repeat: multi-turn chat (history is a growing shared prefix), few-shot prompting, agent loops, and self-consistency sampling where many branches share one context. On the original benchmarks this drove up to ~5–6× higher throughput than prior systems on prefix-heavy workloads.

What does RadixAttention actually reuse across requests?

  • The model's weights, so they load once
  • The KV cache for token prefixes the requests share
  • The final generated text, as a response cache
  • The tokenizer's vocabulary lookup table
RadixAttention caches and reuses the KV cache (the attention keys/values) for shared token prefixes. Weights are always loaded once regardless — that's not the trick. And it is not a response cache: two requests with the same system prompt but different questions still generate different, fresh answers; only the overlapping prefix computation is shared. That's why it speeds things up without changing outputs.

Why the industry runs it

RadixAttention is the headline, but SGLang's adoption rests on a stack of throughput wins that all aim at the same goal: keep the GPU busy, waste no compute.

  • RadixAttention prefix reuse — the big one for chat, few-shot, and agents (above).
  • Continuous batching. New requests slot into the running batch the moment a sequence finishes, instead of waiting for the whole batch — GPU utilization stays high (reported >90%) under mixed, streaming traffic.
  • Zero-overhead scheduler (from v0.4). The CPU prepares the next batch while the GPU is still crunching the current one — it runs "one batch ahead" so the GPU rarely stalls waiting on Python. This alone added roughly ~1.1× throughput on top of everything else.
  • Fast structured outputs. Integrated grammar-constrained decoding (via XGrammar) enforces JSON-schema / regex / EBNF constraints with very low per-token overhead — reported up to ~10× faster than earlier constrained-decoding paths. Handy when a service must emit strict JSON.
  • Serious scale features. Tensor + data parallelism, FP8/INT4 quantization, speculative decoding, and prefill/decode disaggregation for large fleets.
Who runs it

Per the project and 2025–2026 write-ups, SGLang is used in production at scale — reported to power hundreds of thousands of GPUs across organizations including xAI, NVIDIA, AMD, and LinkedIn, and it's a common reference server for launching new open models (Llama, Qwen, DeepSeek, and others), sometimes on day one.

Deep dive — why "day-one model support" matters to you

Because SGLang carries its own model implementations and moves fast, newly released open weights frequently get an SGLang recipe immediately, with the vendor's own recommended launch flags. If your company wants to adopt the latest open model quickly, that release cadence is a practical reason to keep SGLang in the toolbox even if you standardize on something else for older models.

Pros, cons, and SGLang vs vLLM

You will inevitably compare SGLang to vLLM — they're the two dominant open engines and they solve the same problem with different emphases. vLLM pioneered PagedAttention (KV cache as OS-style paged blocks); SGLang layers RadixAttention (prefix reuse) on top of block storage and pairs it with the co-designed frontend. Neither is strictly "faster" — it depends on the workload.

DimensionSGLangvLLM
Prefix-heavy workloads (chat history, shared system prompt, few-shot, agents) Strongest — RadixAttention reuse shines Good (has prefix caching too), usually a step behind on these
Reported throughput, small models (7–8B, H100) Reported edge (order of ~+30% in some benchmarks) Close; workload-dependent
Throughput on very large models (70B+) Roughly comparable — gap narrows to a few % Roughly comparable
Structured / grammar-constrained output A strength (XGrammar, low overhead) Supported; historically higher overhead
Hardware breadth NVIDIA + AMD (ROCm); narrower Broadest — also TPU, Trainium/Inferentia, Gaudi, etc.
Ecosystem / community size / API maturity Large and growing Largest, most integrations and docs
Multi-step / parallel LLM programs First-class frontend DSL Not a focus (serving engine only)

Pick SGLang when: your traffic is prefix-heavy (a fixed system prompt across many questions — your case), multi-turn chat, agents/tool-loops, heavy structured-output needs, or you're chasing peak throughput on NVIDIA/AMD and want the newest models fast.

Lean vLLM when: you need non-NVIDIA/AMD hardware (TPU/Trainium/Gaudi), you value the biggest ecosystem and the most mature/stable APIs, or your prompts are mostly unique with little shared prefix (the RadixAttention edge shrinks toward parity).

Honest take

Benchmark deltas are real but workload- and version-specific — both projects ship improvements monthly and leapfrog each other. Don't over-index on a single "X% faster" number from a blog. For your helpdesk bot, the deciding factor is the shared-prefix win, which is structural, not a benchmark artifact. Prototype on both with your prompts and model and measure TTFT and tokens/sec before committing.

Your future workload is mostly one-off, all-unique prompts (each user pastes a large, different document) with almost no shared prefix. What should you expect?

  • SGLang's RadixAttention still gives a huge speedup because it caches responses
  • SGLang's prefix-reuse advantage largely disappears; the two engines land close
  • vLLM cannot serve this workload at all
  • RadixAttention will slow SGLang down below vLLM
RadixAttention only helps when requests share a prefix. All-unique prompts have nothing to reuse, so its headline advantage evaporates and performance comes down to the general engine (kernels, batching, scheduler) — where the two are close. It's not a response cache, so it can't "recognize" unrelated documents. And the tree lookup is cheap: on a miss it simply falls back to computing everything, so it won't drag SGLang below vLLM. Lesson: match the engine's strength to your actual traffic shape.

Hands-on: stand up a server

The whole path from zero to an OpenAI-compatible endpoint is three moves: install, launch, call.

1 · Install

# needs an NVIDIA (or AMD/ROCm) GPU, recent CUDA, Python 3.9+
pip install --upgrade pip
pip install "sglang[all]"

# or run the official Docker image (pins CUDA + kernels for you):
# docker run --gpus all -p 30000:30000 lmsysorg/sglang:latest ...

Tip: pin a version ("sglang[all]==<x.y.z>") in production and check the model vendor's README for the exact recommended flags — this space moves fast.

2 · Launch the server

python -m sglang.launch_server \
  --model-path Qwen/Qwen2.5-7B-Instruct \
  --host 0.0.0.0 --port 30000
# serves an OpenAI-compatible API at http://localhost:30000/v1
# RadixAttention prefix caching is ON by default — nothing to configure

3 · Call it (curl, then the OpenAI SDK)

curl http://localhost:30000/v1/chat/completions \
  -H "Content-Type: application/json" \
  -d '{"model":"Qwen/Qwen2.5-7B-Instruct",
       "messages":[{"role":"user","content":"What are our office hours?"}]}'
# Python — your existing OpenAI client, just repoint base_url
from openai import OpenAI
client = OpenAI(base_url="http://localhost:30000/v1", api_key="none")

r = client.chat.completions.create(
    model="Qwen/Qwen2.5-7B-Instruct",
    messages=[
        {"role":"system","content":"You are the ACME internal helpdesk."},
        {"role":"user","content":"How do I reset my VPN password?"},
    ])
print(r.choices[0].message.content)

That's it — a running fast-inference server. The system prompt above becomes a shared prefix the moment your second request reuses it.

Key launch flags you'll actually reach for

FlagWhat it does
--model-pathHF model id or local path to serve
--tp-size NTensor-parallel across N GPUs (split one model over N cards for a big model)
--dp-size NData-parallel: N independent replicas for more throughput
--mem-fraction-staticFraction of GPU memory for weights + KV pool. Lower it if you hit OOM.
--context-lengthMax sequence length (raise for long docs, at a memory cost)
--quantizationfp8 / awq / gptq — fit a bigger model on the same card
--api-keyRequire a bearer token on the endpoint
--served-model-nameThe model string clients send (alias the long HF id)

Deep dive — more flags & the two knobs that matter for your box

  • --disable-radix-cache — turn prefix caching off (benchmarking, or a purely unique-prompt workload). Usually leave it on.
  • --chunked-prefill-size — split long prefills into chunks so a giant prompt doesn't block decode for everyone (smoother tail latency).
  • --enable-torch-compile — extra kernel fusion; longer startup, faster steady state.
  • --schedule-conservativeness, --max-running-requests — cap concurrency if you're seeing memory pressure.

On a single modest GPU, the two levers that unstick most setups are --mem-fraction-static (down) and --context-length (down) — both free KV memory. Reach for quantization when the model simply won't fit.

Deep dive — the frontend language & structured output

        
Beyond the OpenAI API, SGLang's DSL lets you script multi-step, branching LLM programs whose
          branches automatically share cached prefixes. sgl.select and grammar constraints mask
          the decode so output is guaranteed valid — useful for triage/classification:

        
import sglang as sgl

@sgl.function
def triage(s, ticket):
    s += sgl.system("You are an IT helpdesk triage bot.")
    s += sgl.user(ticket)
    # constrained choice: decoding is masked to exactly these options
    s += sgl.assistant("Category: " + sgl.select(
            "cat", choices=["password", "vpn", "hardware", "other"]))
    s += sgl.user("Write a one-line reply.")
    s += sgl.assistant(sgl.gen("reply", max_tokens=64, stop="\n"))

sgl.set_default_backend(sgl.RuntimeEndpoint("http://localhost:30000"))
out = triage.run(ticket="I'm locked out and forgot my password.")
print(out["cat"], "->", out["reply"])   # e.g. password -> Use the self-serve reset portal…

        
s.fork(n) spawns n parallel branches that share the prompt's KV cache —
          self-consistency or multi-draft sampling where the shared context is computed once. For strict
          JSON without the DSL, pass a schema straight to the runtime:

        
import requests
r = requests.post("http://localhost:30000/generate", json={
    "text": "Give two colors as a JSON array:",
    "sampling_params": {
        "max_new_tokens": 64,
        "json_schema": '{"type":"array","items":{"type":"string"}}'}})
print(r.json()["text"])   # output is guaranteed to match the schema

Where people go wrong

      
Pitfall
        
Expecting RadixAttention to cache answers. It caches prompt-prefix
          computation, not responses. Two identical questions still each run a full generation; you
          only save the shared-prefix prefill. If you want response-level caching, that's a separate layer you
          add yourself.

      
Pitfall
        
Breaking your own prefix. The cache keys on the exact leading tokens. If
          you jam a timestamp, a request id, or a per-user string at the top of every prompt, no two
          requests share a prefix and the win vanishes. Keep the stable, shared content (system prompt,
          few-shot) first; put anything variable at the end.

      
Pitfall
        
Reading a benchmark as destiny. "SGLang is 30% faster" is true for some model,
          some hardware, some traffic, some version. Yours may differ. Measure TTFT and tokens/sec on your
          own prompts before you standardize.

      
Pitfall
        
Under-provisioning KV memory, then blaming the model. Startup OOM and
          mid-run evictions are usually a --mem-fraction-static / --context-length
          tuning issue, not a bug. Right-size the KV pool for your concurrency and sequence lengths.

    


    
    

      
Cheat-sheet

      
What: open-source LLM serving runtime (OpenAI-compatible) + a Python DSL for
        structured/parallel generation. Signature feature: RadixAttention — reuse the KV
        cache of shared prompt prefixes via a radix tree (LRU-evicted, cache-aware scheduled).

      
Why it's fast: prefix reuse · continuous batching · zero-overhead scheduler
        (runs one batch ahead) · fast grammar-constrained JSON (XGrammar) · TP/DP · FP8/INT4 quantization.
