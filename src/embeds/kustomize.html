Kustomize, from zero to overlays

You already know Deployments, Services, and ConfigMaps. Kustomize is the thin layer that lets you keep one honest set of those manifests and reshape them per environment — no templating language, no variables, just plain YAML in and plain YAML out. This module is the mental model plus the exact syntax to make it do what you want.

Full mode — deep-dive blocks visible

The one idea

Kustomize is a pure function on YAML: it reads a directory of valid Kubernetes manifests plus a recipe file (kustomization.yaml) and prints a new stream of valid manifests to stdout. It never runs your cluster, never stores state, and — this is the part that trips up people coming from Helm — it has no template language. There are no {{ .Values }} placeholders, no if, no range. The input manifests are already deployable as-is; Kustomize just transforms them.

Key insight

Helm parameterizes — you write templates with holes and fill them at install time. Kustomize overlays — you keep a real, complete base and stack declarative edits (patches and transformers) on top of it. The base stays deployable on its own; each environment is "the base, plus this small diff."

Everything else in this module is just the vocabulary for describing those edits precisely: which resources to include, what to rename, what to patch, and what to generate.

How you run it

Kustomize ships inside kubectl, and also as a standalone binary. Two commands cover almost everything:

# Render the manifests to stdout — inspect, diff, pipe. Changes nothing.
kubectl kustomize ./overlays/prod
# same thing with the standalone binary:
kustomize build ./overlays/prod

# Render AND apply in one step (the -k flag = "kustomize this dir"):
kubectl apply -k ./overlays/prod

The mental loop is: edit YAML → kubectl kustomize to see the rendered output → apply. Because the output is just manifests, you can pipe it into kubectl diff -f -, into a GitOps tool (Argo CD and Flux both call kustomize build under the hood), or into grep. That "it's only a renderer" property is the whole reason it composes so well.

Watch the version

The kubectl-embedded Kustomize often lags the standalone binary by several releases. If a field works in kustomize build but not kubectl kustomize, your kubectl is old. For anything nontrivial, install the standalone kustomize and check kustomize version.

You run kubectl kustomize ./overlays/prod. What happens to your cluster?

  • The prod overlay is applied and pods start rolling.
  • Nothing — the rendered manifests are printed to stdout; the cluster is untouched.
  • Kustomize connects to the cluster to diff current vs desired state.
kubectl kustomize (and kustomize build) are build commands: pure YAML→YAML, no cluster contact. You need kubectl apply -k (or apply -f - on the piped output) to actually change anything. Keeping build and apply separate is what lets you review the diff first.

Anatomy of a kustomization.yaml

Every Kustomize directory has exactly one file named kustomization.yaml (or .yml/Kustomization). It's the recipe. Its top-level keys fall into three buckets, and once you can name the bucket a key belongs to, the whole tool clicks:

  • Inputs — resources, components, configMapGenerator, secretGenerator: what manifests enter the pipeline.
  • Transformers — namespace, namePrefix/nameSuffix, labels, commonAnnotations, images, replicas: broad, structured edits applied across everything.
  • Patches — patches, replacements: surgical, targeted edits.
apiVersion: kustomize.config.k8s.io/v1beta1
kind: Kustomization

# --- inputs ---
resources:
  - deployment.yaml
  - service.yaml

# --- transformers ---
namespace: web-prod
namePrefix: prod-
labels:
  - pairs:
      app.kubernetes.io/part-of: storefront
    includeSelectors: false
images:
  - name: nginx
    newTag: 1.27.3

# --- patches ---
patches:
  - path: replicas-patch.yaml

That's the whole shape. The rest of this module walks each bucket with the syntax you'd actually type.

Inputs: resources & bases

resources is a list of things to pull into the build. Each entry is one of:

  • a local manifest file — deployment.yaml
  • a directory that itself contains a kustomization.yaml — this is how an overlay includes a base: ../../base
  • a remote reference — a Git URL like github.com/org/repo/config?ref=v1.4.0 (always pin ?ref=).
Deprecated: bases

Old tutorials use a separate bases: field. It was deprecated years ago — directories now go in resources alongside files. If you inherit a repo with bases:, run kustomize edit fix to migrate it automatically.

A base is just a normal Kustomize directory that happens to be referenced by others. There's nothing special marking it as "a base" — the role is relational. This is the directory layout you'll see everywhere:

app/ ├── base/ │ ├── kustomization.yaml # resources: [deployment.yaml, service.yaml] │ ├── deployment.yaml │ └── service.yaml └── overlays/ ├── dev/ │ ├── kustomization.yaml # resources: [../../base] + dev tweaks │ └── replicas-patch.yaml └── prod/ ├── kustomization.yaml # resources: [../../base] + prod tweaks └── replicas-patch.yaml

You build an overlay (overlays/prod), never the base directly for deployment. The overlay says resources: [../../base] and then layers its environment-specific edits.

Key insight

An overlay never edits the base's files. It composes on top of them. That's the property that keeps a 3-environment repo honest: there is exactly one copy of the real manifest, and each environment is a readable diff against it — not a forked, drifting copy.

Transformers: broad edits

Transformers are top-level fields that sweep across every resource in the build and apply a structured change. You don't target them at one object; they're the "do this to everything" knobs. The workhorses:

FieldWhat it does
namespaceSets metadata.namespace on all namespaced resources, and rewrites references (e.g. a RoleBinding's subject namespace).
namePrefix / nameSuffixPrepends/appends to every metadata.name, and rewrites references to those names (a Service referenced by a Deployment follows along).
labelsAdds labels to metadata.labels on every resource. includeSelectors controls whether they're also added to selectors/pod templates.
commonAnnotationsAdds annotations to every resource's metadata.annotations.
imagesOverrides container image name / newName / newTag / digest without editing the Deployment YAML.
replicasSets spec.replicas by resource name — the clean way to scale per environment.
 
images:
  - name: nginx              # match containers using image "nginx[:...]"
    newName: registry.internal/nginx
    newTag: 1.27.3
  - name: api
    digest: sha256:abc123…   # pin by digest instead of tag

replicas:
  - name: web                # matches metadata.name BEFORE namePrefix is applied
    count: 5

Deep dive — the labels footgun and why commonLabels was replaced

The old commonLabels field always injected your labels into pod-template selectors too. That sounds convenient until you realize a Deployment's spec.selector is immutable — so adding a common label to an already-deployed Deployment made the next apply fail outright. The replacement, labels, defaults to includeSelectors: false: labels land on metadata only, selectors are left alone. You opt into selector injection explicitly, per label group:

labels:
  - pairs:
      app.kubernetes.io/part-of: storefront
    includeSelectors: false   # metadata only — safe on live Deployments
  - pairs:
      app: web
    includeSelectors: true    # also selector + pod template — set on NEW resources only

Rule of thumb: use includeSelectors: true only on brand-new resources where the selector doesn't exist yet; leave it false for org-wide labels you sprinkle on existing workloads.

Deprecated: commonLabels

commonLabels: {app: web} still works but warns. It maps to a labels entry with includeSelectors: true. Prefer the explicit labels list.

Your Deployment is already live. You add commonLabels: {team: payments} and kubectl apply -k. It fails with an error about the selector. Why?

  • Common labels must be unique per resource.
  • commonLabels injects into the immutable pod selector, and you can't change a live Deployment's selector.
  • Labels can't contain a / or the apply is rejected.
A Deployment's spec.selector.matchLabels is immutable after creation. commonLabels (and labels with includeSelectors: true) write into it, so the apply is rejected. Use the newer labels with includeSelectors: false to add metadata labels without touching the selector.

Patches: surgical edits

Transformers handle "rename everything" and "set every image." Patches are for the specific edits — bump this Deployment's replicas, add an env var to that container, drop a volume. The modern field is a single list called patches, and each entry is:

  • a patch body, given inline (patch:) or as a file (path:), and
  • an optional target selector saying which resource(s) it hits.

Kustomize auto-detects the two patch flavors by shape:

1. Strategic Merge Patch (SMP)

You write a partial manifest — just enough of the resource to identify it and the fields you want to change. Kustomize deep-merges it into the base. This is the one you'll reach for 90% of the time because it reads like the resource itself:

# replicas-patch.yaml — a partial Deployment
apiVersion: apps/v1
kind: Deployment
metadata:
  name: web          # identifies the target
spec:
  replicas: 3         # the only thing that changes
# in kustomization.yaml — no target needed; the patch's own kind+name identify it
patches:
  - path: replicas-patch.yaml

Lists merge intelligently: containers merge by name (the merge key baked into the Kubernetes type), so a patch that names one container edits only that one and leaves the siblings alone. To force other behavior, SMP has directives: $patch: replace (swap the whole list/map instead of merging) and $patch: delete (remove an element).

2. JSON 6902 Patch

An RFC 6902 patch is an explicit list of operations — op, path, value. It's precise and unambiguous, which makes it the right tool when strategic merge is fuzzy: removing one item from a list by index, or editing a field on a resource kind Kustomize has no merge-key knowledge of (a CRD).

patches:
  - target:                 # JSON6902 needs a target selector
      kind: Deployment
      name: web
    patch: |-
      - op: replace
        path: /spec/replicas
        value: 3
      - op: add
        path: /spec/template/spec/containers/0/env
        value:
          - {name: TIER, value: prod}

The target selector — patch many objects at once

Give a patch a target and it applies to every resource that matches. This is how you say "add this security context to all Deployments" without naming each one. A target can filter on group, version, kind, name (regex), namespace, labelSelector, and annotationSelector:

patches:
  - target:
      kind: Deployment
      labelSelector: "tier=backend"
    patch: |-
      - op: add
        path: /spec/template/spec/securityContext
        value: {runAsNonRoot: true}
Deprecated: patchesStrategicMerge & patchesJson6902

These two older fields have been folded into the single patches list, which auto-detects the flavor. You'll see the old ones constantly in existing repos — recognize them, but write new code with patches. kustomize edit fix migrates.

Generators: ConfigMaps & Secrets

      
You could hand-write a ConfigMap manifest, but Kustomize can generate one from
        literals, files, or an env file — and it does something clever with the name:

      
configMapGenerator:
  - name: app-config
    literals:
      - LOG_LEVEL=info
      - TIMEOUT=30s
    files:
      - nginx.conf          # key = filename, value = file contents

secretGenerator:
  - name: db-creds
    envs:
      - db.env             # KEY=VALUE lines, base64-encoded for you


      
The generated object is named app-config-t4f8h9k2c6 — a content hash is
        appended. Change any value and the hash changes, producing a new ConfigMap name. And
        here's the magic: Kustomize rewrites every reference to app-config in your
        Deployments to the hashed name.


      
Key insight
        
The hash suffix isn't cosmetic — it's how Kustomize gives you automatic rolling restarts
          on config change. New config → new ConfigMap name → the Deployment's pod spec now
          references a different name → Kubernetes sees a changed pod template → rolling update. Without
          it, editing a ConfigMap's data leaves running pods stuck on the old values.


      
Control it with generatorOptions and behavior:

      
generatorOptions:
  disableNameSuffixHash: true   # stable name, e.g. for a CM other tools read by name

configMapGenerator:
  - name: app-config
    behavior: merge          # merge into a same-named generator in the BASE
    literals:
      - LOG_LEVEL=debug      # overlay overrides just this key

      
behavior is create (default), merge (add/override keys of
        a base generator), or replace (swap it wholesale) — the overlay's way to tweak a
        base-defined ConfigMap.


      
Pitfall
        
If something outside Kustomize's view references the ConfigMap by its plain name
          (a hardcoded name in an app arg, a separately-applied resource), the hash suffix breaks that
          link — Kustomize only rewrites references it can see in the build. Either include that resource
          in the same build so the rewrite reaches it, or set disableNameSuffixHash: true.


      

        
Why does configMapGenerator append a hash like -t4f8h9k2c6 to the name?

        
    
          
  • To avoid name collisions between namespaces.
    • 
          
  • So a config change yields a new name, which changes the pod template and triggers a rolling update.
    • 
          
  • It's a checksum Kubernetes verifies on apply.
    • 
        

        
The hash makes config changes propagate. Mutating a ConfigMap's
          data in place does not restart pods — they keep the mounted/old values. By minting a new
          ConfigMap name on every change and rewriting references to it, Kustomize forces the Deployment's
          pod spec to change, and Kubernetes rolls the pods. Turn it off with
          disableNameSuffixHash only when you specifically need a stable name.

Deep dive — ordering caveats

        
The real ordering has subtleties: generators run before transformers, namePrefix/
          nameSuffix and namespace also rewrite references (not just
          the names themselves), and patches apply near the end so they see already-transformed
          resources. One practical consequence: the replicas transformer and a patch's
          target: name match against the original name, before namePrefix
          is applied — target the base name, not the prefixed one. When in doubt, run
          kustomize build and read the output; it's the ground truth.

      

    


    
    

      
Replacements & components

      
Two more pieces you'll meet in mature repos, both replacing older mechanisms.


      
replacements — copy a value from A to B

      
Sometimes one value must appear in several places kept in sync — an image tag that also goes in
        a label, a Service name that a ConfigMap references. The old vars field did this and
        was fragile; replacements is the current, well-defined way. It reads a value from a
        source field and writes it into one or more target field paths:

      
replacements:
  - source:
      kind: Deployment
      name: web
      fieldPath: spec.template.spec.containers.0.image
    targets:
      - select:
          kind: ConfigMap
          name: build-info
        fieldPaths:
          - data.image

      
Unlike vars, replacements don't need the value to look like a string token in the
        target — they just overwrite the field. That's why they're reliable where vars
        silently failed.


      
components — reusable, opt-in mixins

      
A Component is a mini-kustomization (kind: Component) that an overlay can
        pull in via components:. Unlike a base (which is a complete app), a component is a
        fragment — "the monitoring sidecar," "the mTLS patch" — that several overlays share.
        Use it when the same optional feature is toggled on across environments:

      
# components/mtls/kustomization.yaml
apiVersion: kustomize.config.k8s.io/v1alpha1
kind: Component
patches:
  - path: inject-sidecar.yaml

      
# overlays/prod/kustomization.yaml
resources:
  - ../../base
components:
  - ../../components/mtls    # prod gets mTLS; dev overlay simply omits it

      
Components apply in list order and can add resources, patches, and generators — think of them as
        composable feature flags for your manifests.

    


    
    

      
Where people go wrong

      
Expecting templates
        
The number-one misconception: reaching for Kustomize when you actually want
          parameterization (loops, conditionals, a value computed from three inputs). Kustomize
          has none of that on purpose. If your need is "render N near-identical objects from a list" or
          "branch on a flag," that's Helm — or Helm rendered and then Kustomized. Kustomize
          shines at "one base, small per-env diffs," not at generating structure.


      
Silent patch no-ops
        
A patch whose target matches nothing simply does nothing — no error. Get the
          group/version/kind/name wrong (e.g. targeting
          Deployment in the wrong apiVersion, or the prefixed name instead of the
          base name) and your change vanishes with no warning. Always diff the build output.


      
namespace on cluster-scoped resources
        
The namespace transformer only touches namespaced resources. ClusterRoles,
          ClusterRoleBindings, PVs, and the like are left alone — as they must be. Don't expect it to
          "move" a cluster-scoped object.


      
Legacy fields everywhere
        
You'll open real repos full of bases, patchesStrategicMerge,
          patchesJson6902, commonLabels, and vars. They still work
          (with warnings) but are the old syntax. kustomize edit fix rewrites a
          kustomization.yaml to the current fields automatically — run it before you start editing.
