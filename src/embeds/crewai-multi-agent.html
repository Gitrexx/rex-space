The one idea

CrewAI models an AI workflow as a team of role-playing agents. Instead of one giant prompt doing everything, you give each LLM a role ("Senior Research Analyst"), a goal, and a backstory, hand it some tools, then assign it discrete tasks. A crew runs those tasks in a defined order and threads each task's output into the next. If humans-as-a-team is your mental model, CrewAI is that model turned into a Python object graph.

Key insight

CrewAI ships two orchestration primitives, and the whole framework makes sense once you separate them. Crews = autonomous, role-based collaboration (agents decide how to reach a shared goal). Flows = deterministic, event-driven control (you decide exactly what fires when, with state, branching, and persistence). Real systems combine them: a Flow orchestrates the plumbing and calls Crews for the "let the agents figure it out" steps.

The building blocks

Four objects carry almost everything. Learn these and you can read any CrewAI codebase.

  • Agent — a configured LLM persona: role, goal, backstory, plus tools, an llm, and knobs like allow_delegation and max_iter.
  • Task — a unit of work: a description, an expected_output (this is load-bearing — it steers the model), the agent that owns it, optional tools, and optional context (outputs of earlier tasks to feed in).
  • Crew — the team: a list of agents, a list of tasks, and a process. You start it with crew.kickoff(inputs=...).
  • Process — the execution strategy: sequential (default; tasks run top-to-bottom), or hierarchical (a manager LLM plans and delegates). A consensual process is on the roadmap, not shipped.

Real CrewAI — a minimal crew (run this in your own env, needs an API key)

from crewai import Agent, Task, Crew, Process

researcher = Agent(
    role="Senior Research Analyst",
    goal="Find the 3 most important facts about {topic}",
    backstory="You are meticulous and cite sources.",
    llm="gpt-4o",            # any LiteLLM-supported model id
    allow_delegation=False,
    verbose=True,
)
writer = Agent(
    role="Technical Writer",
    goal="Turn research into a tight 5-bullet brief",
    backstory="You write for busy engineers.",
    llm="gpt-4o",
)

research = Task(
    description="Research {topic}. Focus on facts an engineer would act on.",
    expected_output="3 bullet points, each with a one-line source.",
    agent=researcher,
)
write = Task(
    description="Write the brief from the research.",
    expected_output="Exactly 5 bullets, no preamble.",
    agent=writer,
    context=[research],     # research output is injected here
)

crew = Crew(
    agents=[researcher, writer],
    tasks=[research, write],
    process=Process.sequential,
    verbose=True,
)

result = crew.kickoff(inputs={"topic": "vector databases"})
print(result.raw)           # the final string
print(result.token_usage)   # UsageMetrics: prompt/completion/total tokens

Deep dive — the execution model & CrewOutput

In a sequential crew, tasks execute in list order. The output of each task is automatically available to the next as context, and you can also wire it explicitly with context=[task_a, task_b]. Each agent runs its own ReAct-style loop internally — reason, optionally call a tool, observe, repeat — capped by max_iter. kickoff() returns a CrewOutput: .raw (string), .pydantic / .json_dict (when a task sets output_pydantic / output_json), .tasks_output (per-task results), and .token_usage.

hierarchical is different: you do not pin tasks to agents up front. You supply a manager_llm (or a full manager_agent), and the manager decides which worker handles each task, in what order, and when the goal is met. More autonomy, more tokens, more ways to go wrong — see Pitfalls.

There's also a @CrewBase project layout the CLI scaffolds (crewai create crew my_project): agents and tasks live in config/agents.yaml and config/tasks.yaml, and Python glues them with @agent, @task, and @crew decorators. The YAML keeps prompts out of code and is the recommended shape for anything beyond a toy.

In the sequential crew above, how does the Writer task see the Researcher's findings?

  • They share one global variable you must populate by hand
  • The previous task's output is threaded in as context (here, explicitly via context=[research])
  • The Writer re-runs the research itself from the {topic} input
  • It doesn't — each task is fully isolated
A sequential process passes each task's output forward as context; context=[research] makes that dependency explicit and deterministic. Nothing is copied by hand, and tasks are not isolated — the whole point of a crew is the hand-off.

Where Flows come in

A Crew is stateless and single-shot: kickoff() runs once and returns. That's perfect for "research → analyze → write", but it can't branch on a result, loop with retries, pause for human approval, or resume after a crash. That's what Flows add. A Flow is a Python class whose methods are wired into an event graph with decorators:

  • @start() — an entry point; runs when the flow kicks off.
  • @listen(x) — fires when method x (or an emitted label) completes.
  • @router(x) — like listen, but returns a label to choose which branch fires next.
  • or_(a, b) / and_(a, b) — fire when any / all of several steps finish.
  • @persist — checkpoint the flow's state (SQLite-backed) so it can resume.

Real CrewAI — a Flow with typed state, a router, and a crew call

from crewai.flow.flow import Flow, start, listen, router
from pydantic import BaseModel

class GuideState(BaseModel):     # typed state = validated + autocompleted
    topic: str = ""
    quality: int = 0
    article: str = ""

class GuideFlow(Flow[GuideState]):
    @start()
    def outline(self):
        self.state.topic = "multi-agent systems"
        # ... could call a Crew here and store its .raw on self.state

    @listen(outline)
    def draft(self):
        self.state.article = "first draft ..."
        self.state.quality = 4   # imagine a critic agent scored it

    @router(draft)
    def gate(self):
        return "ship" if self.state.quality >= 3 else "revise"

    @listen("ship")
    def publish(self):
        return "published: " + self.state.article

    @listen("revise")
    def rewrite(self):
        return "looping back to draft ..."

result = GuideFlow().kickoff()   # walks the event graph, threads state

Deep dive — structured vs unstructured state, and when to persist

Flow[GuideState] gives you structured state: a validated Pydantic model with editor autocomplete and a runtime guarantee on shape. Drop the type parameter (class GuideFlow(Flow):) and you get unstructured state — a dict-like self.state you can throw anything into. Structured is the right default for anything you'll maintain; unstructured is fine for a spike.

@persist writes state to a local store after each step, keyed by a flow id, so a long or human-in-the-loop flow survives a restart and can be resumed. This is the feature people reach for when they discover a bare Crew has no checkpointing.

You need a long-running pipeline with a human-approval gate, retry-on-failure, and state you can persist and resume. Which CrewAI primitive fits?

  • A hierarchical Crew — let the manager handle it
  • A Flow with @start/@listen/@router and @persist
  • A single Agent with a very detailed backstory
  • Process.consensual
Crews are stateless and single-shot — no branching, retries, pausing, or persistence. Flows are the event-driven layer built for exactly that: @router branches, @persist checkpoints, and gates/loops are just methods. A better manager LLM doesn't give a Crew durability, and consensual isn't shipped.

When to reach for it

      
CrewAI's sweet spot is multi-step work that decomposes into clear roles,
        where you want to ship fast and the abstraction ("a team of specialists") matches the
        problem.


      
Good fits

      
    
        
  • Research → analysis → synthesis pipelines: market/lead research,
          competitive briefs, literature scans, report generation.
    • 
        
  • Content production with distinct hats: outline → draft → edit → SEO,
          each a role with its own tools.
    • 
        
  • Structured automation with a handful of steps and tool calls
          (search, scrape, SQL, code exec) that a junior engineer should be able to stand up in a day.
    • 
        
  • Rapid prototyping of an agentic idea before committing to heavier
          infrastructure — the fastest framework to a working demo.
    • 
      


      
Reach for Flows (still CrewAI) when

      
You need branching, loops, retries, human-approval gates, or durable state — wrap Crews and
        direct LLM calls in a Flow and keep the control flow explicit.


      
Consider a different tool when

      
    
        
  • You need fine-grained state control, checkpointing, and audit trails for a
          regulated production system → LangGraph.
    • 
        
  • The core is open-ended multi-party dialogue — debate, negotiation,
          consensus among many agents → AutoGen (note: now in maintenance mode).
    • 
        
  • You're all-in on OpenAI and want the thinnest possible layer →
          OpenAI Agents SDK.
    • 
        
  • The task is genuinely one agent with tools — you may not need a crew at all.

CrewAI, honestly — pros & cons

      
Pros: lowest barrier to a working multi-agent app (a crew in ~30–60 lines);
        the role/goal/backstory abstraction maps to how people think about teams; YAML config keeps
        prompts out of code; Crews and Flows cover both autonomy and control; large adoption,
        an active project, and a broad tools ecosystem.

      
Cons: the abstraction hides machinery, so fine-grained debugging and
        agent-to-agent control are harder than in a graph framework; a bare Crew is stateless
        (no checkpointing — that's what Flows are for); hierarchical mode needs a capable (read:
        pricier) manager LLM and can loop or over-delegate; token/cost can balloon; error handling is
        coarse. LangGraph overtook it in GitHub stars in early 2026 on the strength of its
        production-control story — though CrewAI Flows narrow much of that gap.

Where people go wrong


      
Pitfall
        
Expecting a Crew to be durable. A Crew runs once and holds no
          checkpoints — crash halfway and you restart from zero, re-paying every token. Anything
          long-running or human-gated belongs in a Flow with @persist.


      
Pitfall
        
Token & cost blow-up. Every agent step is one or more LLM calls;
          hierarchical adds a manager on top; context accumulates down the pipeline. Scope tasks
          tightly, cap max_iter, give cheap models to cheap roles, and prefer structured
          outputs over "write me everything".


      
Pitfall
        
Over-trusting delegation. allow_delegation=True plus a weak
          manager LLM leads to agents delegating in circles, re-doing work, or inventing hand-offs.
          Keep delegation off unless you need it, and give the manager a strong model.


      
Pitfall
        
Vague expected_output. It's not a comment — it actively steers
          the model. "A helpful summary" yields mush; "Exactly 5 bullets, no preamble, each ≤ 20 words"
          yields something testable. For machine-readable results use output_pydantic /
          output_json.


      
Pitfall
        
Treating it as deterministic. It's LLM-driven — the same inputs can vary.
          Test on shape (schema, bullet count) not exact text, pin structure with Pydantic
          outputs, and put real control flow in Flows rather than hoping agents "just handle it".
