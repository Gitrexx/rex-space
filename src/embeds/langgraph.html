AI Agents · Orchestration Frameworks

LangGraph for Advanced Agent Building

You already ship LLM features — you just haven't built on LangGraph yet. This module skips "what is an agent" and goes straight to the model that matters: an agent is a state machine, and LangGraph is a durable graph runtime for running one. We'll cover state & reducers, cyclic control flow, persistence, human-in-the-loop, and multi-agent orchestration — then brainstorm portfolio-grade projects to practise each of them.

Full mode — deep-dive blocks visible

The one idea

A capable agent is a loop over shared state: think → act → observe → decide whether to go again. That loop has cycles, branches, and points where you want to pause, persist, or hand off. A plain chain (a DAG) can't express a cycle, and an AgentExecutor-style while-loop hides all the interesting control flow inside one opaque function.

LangGraph's answer: model the agent explicitly as a directed graph. Nodes are plain functions that read the current state and return an update. Edges — including conditional ones — decide what runs next, and they're allowed to point backwards, so loops are first-class. A shared, typed State object flows through, and a checkpointer can snapshot it at every step so a run can pause for a human, survive a crash, or resume days later.

Key insight

Everything else in LangGraph — persistence, human-in-the-loop, streaming, multi-agent — falls out of two primitives done well: an explicit graph of nodes and edges, and a state channel with a reducer that says how each node's output merges in. Learn those two and the rest is composition.

Where LangGraph sits (and where it doesn't)

You've seen the higher-level frameworks. LangGraph deliberately lives one layer below them: it is the low-level orchestration engine, not a role/task abstraction. That's the whole trade — more control and durability, more wiring.

Deep dive — LangGraph vs LangChain vs LangGraph Platform

LangChain is the components layer (model wrappers, tools, retrievers, message types). LangGraph is the orchestration/runtime layer that runs those components as a stateful graph — the two are made by the same team and interlock. As of the v1 releases (2026), the modern high-level agent factory create_agent lives in langchain and is itself built on LangGraph, so "use LangChain's agent" and "use LangGraph" are no longer opposing choices — you get a compiled LangGraph graph either way. LangGraph Platform (deployment) + Studio (visual graph debugger) + LangSmith (tracing) are the commercial/ops surface on top; none of them are required to use the open-source library.

Why does LangGraph model an agent as a graph instead of a linear chain?

  • Graphs render more nicely in a UI than chains do.
  • Agents need cycles — reason → act → observe → reason again — and a DAG can't express a backward edge.
  • Because graph traversal is asymptotically faster than running a chain.
The defining behaviour of an agent is iteration: it may call a tool, read the result, and decide to call another before answering. That's a loop. A chain / DAG is acyclic by definition, so it structurally can't loop back. Rendering and speed are not the reason — expressiveness is.

The core primitives: State, Nodes, Edges

Three things and a compile step. Get these and you can read any LangGraph codebase.

1 · State + reducers

State is a typed dictionary (a TypedDict or a Pydantic model). Each key is a channel. A channel can carry a reducer — a function that says how a node's output for that key merges into what's already there. Without a reducer, an update replaces the value; with one, it merges. The canonical example is add_messages, which appends new messages (and updates any with a matching id) instead of overwriting the history.

real langgraph — minimal graph 
# pip install -U langgraph langchain
from typing import Annotated, TypedDict
from langgraph.graph import StateGraph, START, END
from langgraph.graph.message import add_messages

class State(TypedDict):
    # the `add_messages` reducer makes this channel APPEND, not overwrite
    messages: Annotated[list, add_messages]

def chatbot(state: State):
    # a node takes state, returns a PARTIAL update (just the keys it changes)
    return {"messages": [llm.invoke(state["messages"])]}

builder = StateGraph(State)
builder.add_node("chatbot", chatbot)
builder.add_edge(START, "chatbot")
builder.add_edge("chatbot", END)
graph = builder.compile()

graph.invoke({"messages": [{"role": "user", "content": "hi"}]})
# -> {"messages": [HumanMessage("hi"), AIMessage("Hello! ...")]}
Watch the return value

Nodes return a partial update — only the channels they touched — never the whole state. The engine folds each update in through that channel's reducer. This is the single most important habit to internalise.

2 · Nodes

A node is just a function (state) -> partial_update (sync or async). It can call a model, hit a database, run a tool, or do pure Python. Nodes don't know about each other — they only read state and return an update. That isolation is what makes the graph testable and the flow inspectable.

3 · Edges — including conditional ones

add_edge(a, b) is an unconditional "after a, run b". add_conditional_edges(a, router) calls your router(state) function and routes to whichever node name (or END) it returns. That router is where an agent decides "do I need a tool, or am I done?" — and where the loop closes.

real langgraph — the ReAct tool loop, wired by hand 
from langgraph.prebuilt import ToolNode, tools_condition

builder.add_node("tools", ToolNode(tools))          # runs any tool the model asked for
builder.add_conditional_edges("chatbot", tools_condition)  # tool_calls? -> "tools" ; else -> END
builder.add_edge("tools", "chatbot")              # THE CYCLE: after tools, think again
# chatbot -> (tools -> chatbot)* -> END      <-- a ReAct agent is exactly this loop

A node returns {"messages": [new_msg]} and the channel uses the add_messages
          reducer. What happens to the messages already in state?

        
    
          
  • They're replaced — the node's return value becomes the new messages list.
    • 
          
  • They're kept; new_msg is appended (and a message with a matching id would be updated in place).
    • 
          
  • They're cleared, and only messages with a reducer survive.
    • 
        

        
A reducer defines the merge. add_messages appends new messages and updates
          any existing one whose id matches (useful for streaming edits). Return-replaces-value is what you'd get
          without a reducer — exactly the bug that silently wipes conversation history.

      

    


    
    

      
The execution model: super-steps

      
LangGraph runs on a Pregel-style, bulk-synchronous model. Each iteration is a super-step:
        every node on the current frontier runs, their updates are collected, the reducers fold them into state, and
        then edges pick the next frontier. Nodes at the same frontier run in parallel; a fan-out followed by
        a merge is just several nodes writing to a channel whose reducer combines them.

        reducer. Note the two cycles through tools → agent before the conditional edge finally routes to

Deep dive — parallelism, and why a reducer is not optional there

        
Because a whole frontier runs before edges are re-evaluated, two nodes that both write messages
          in the same super-step would collide — which is exactly why that channel needs a reducer to define
          a deterministic merge. Fan-out/fan-in (e.g. query three retrievers in parallel, then combine) is idiomatic:
          add three nodes, edge into all of them from one router, and give the result channel a reducer like
          operator.add. If a node raises, LangGraph doesn't silently drop the super-step — with a
          checkpointer, the completed work is durable and you can resume from the last good snapshot.

      

    


    
    

      
The fast path: create_agent

      
You rarely hand-wire the ReAct loop in production — you reach for the prebuilt and drop to raw
        StateGraph only when you need custom flow. In the v1 era the recommended factory is
        create_agent from langchain.agents. It returns a compiled LangGraph graph, so
        everything below (persistence, streaming, HITL) applies unchanged.


      real langgraph — modern prebuilt agent
      
from langchain.agents import create_agent   # the 2026-recommended entry point

def get_weather(city: str) -> str:
    """Look up the weather for a city."""
    return f"{city}: 18°C, clear"

agent = create_agent(
    model="openai:gpt-4o",      # or an Anthropic / local model handle
    tools=[get_weather],
    system_prompt="You are a concise weather assistant.",
)
agent.invoke({"messages": [{"role": "user", "content": "weather in Paris?"}]})
# agent is a compiled graph: agent.stream(...), .get_state(...), checkpointer=... all work

Persistence & durable execution

      
Compile with a checkpointer and pass a thread_id, and LangGraph snapshots the
        full state at every super-step. That one change unlocks: multi-turn memory (each thread_id is an
        independent conversation with its own history), crash recovery (resume from the last snapshot), and
        time-travel debugging (get_state_history, then re-run from any past checkpoint).


      real langgraph — add memory with one argument
      
from langgraph.checkpoint.memory import InMemorySaver  # dev; swap for Postgres/SQLite in prod

graph = builder.compile(checkpointer=InMemorySaver())
cfg = {"configurable": {"thread_id": "user-42"}}

graph.invoke({"messages": [{"role":"user","content":"my name is Rex"}]}, cfg)
graph.invoke({"messages": [{"role":"user","content":"what's my name?"}]}, cfg)
# -> "Your name is Rex."  (same thread_id => history was reloaded from the checkpoint)

graph.get_state(cfg)            # current StateSnapshot: values + next nodes + config
list(graph.get_state_history(cfg))  # every checkpoint — rewind and branch from any of them
# Production: from langgraph.checkpoint.postgres import PostgresSaver


      
Why this is the headline feature
        
Because state lives in a checkpoint, a paused run holds no compute. An agent can wait minutes,
          hours, or days for a human or an external event and resume "as if milliseconds had passed." That's what
          makes long-running, human-gated agents practical — and it's the foundation the next section stands on.

Human-in-the-loop

      
Call interrupt(payload) inside a node. The graph pauses, persists its state
        (this is why it needs a checkpointer), and surfaces payload to the caller. Whenever the human
        responds — seconds or days later — you resume by invoking with Command(resume=value), and
        interrupt(...) returns that value as if it had never stopped.


      real langgraph — approve/reject before a risky action
      
from langgraph.types import interrupt, Command

def review(state):
    decision = interrupt({"action": state["proposed_action"],
                          "question": "Approve this refund?"})   # <-- pauses here
    if decision == "approve":
        return {"status": "executed"}
    return {"status": "cancelled"}

graph = builder.compile(checkpointer=InMemorySaver())
cfg = {"configurable": {"thread_id": "ticket-7"}}

graph.invoke(inputs, cfg)     # runs until interrupt(); state is saved, control returns to you
# ... hours later, after a human clicks "approve" in your UI ...
graph.invoke(Command(resume="approve"), cfg)   # interrupt() returns "approve"; the node finishes


      
Common HITL shapes: approve/reject a tool call, edit the proposed state
        before continuing, review & correct a model output, or an explicit ask-a-human
        tool the agent can choose to call. There are also static breakpoints — compile(interrupt_before=["tools"])
        — for pausing without editing node code.


      

        
Your interrupt() call raises or the resume never picks up state. What's the most likely cause?

        
    
          
  • You forgot to wrap the node in a try/except.
    • 
          
  • You compiled without a checkpointer (or didn't pass a thread_id), so there's nowhere to persist and resume from.
    • 
          
  • interrupt() only works inside conditional edges, not nodes.
    • 
        

        
Interrupts are built on persistence: the state must be saved to a checkpoint so the run
          can stop and later resume against the same thread_id. No checkpointer (or no thread id) means
          no snapshot — the pause/resume machinery has nothing to stand on. interrupt() lives in node
          code, not in edges.

      

    


    
    

      
Streaming (so the UI isn't a spinner)

      
Agents are slow; you stream to keep them feeling alive. graph.stream(inputs, cfg, stream_mode=...)
        gives you different granularities, and you can pass a list to get several at once:

      
    
        
  • "values" — the full state after each super-step.
    • 
        
  • "updates" — just the per-node delta each step (great for logging progress).
    • 
        
  • "messages" — LLM tokens as they're generated (the ChatGPT-style typewriter).
    • 
        
  • "custom" — whatever you emit from inside a node via the stream writer.
    • 
      

      
for chunk in graph.stream(inputs, cfg, stream_mode="updates"):
    print(chunk)   # {'agent': {...}}  then  {'tools': {...}}  then ...

    


    
    

      
Multi-agent orchestration

      
Once one agent works, scaling to several is a routing question. LangGraph gives you two shapes plus a way to
        nest them. Any compiled graph can also be dropped in as a node inside a bigger graph (a
        subgraph), so an "agent" and a "team" are the same primitive at different scales.

      

        
PatternHowStrengthCost

        
          
SupervisorOne router node delegates to worker agents and synthesises their outputs (langgraph-supervisor).Every decision is visible in one place — easy to reason about and trace. The production default.Extra LLM hop through the supervisor on each turn.

          
SwarmPeer agents hand off directly to each other via handoff tools that return a Command (langgraph-swarm).Fewer LLM calls, faster — no middle-man.Control flow is emergent; you lean harder on tracing to follow it.

          
HierarchicalSupervisors nested as workers under a top-level supervisor.Scales past ~6 workers by keeping any one router's fan-out small.More moving parts; deeper traces.

        
      


      real langgraph — a supervisor over two specialists
      
from langchain.agents import create_agent
from langgraph_supervisor import create_supervisor

researcher = create_agent(model, tools=[web_search], name="researcher")
coder      = create_agent(model, tools=[run_python], name="coder")

app = create_supervisor(
    agents=[researcher, coder],
    model=model,
    prompt="Route research questions to researcher, code to coder, then summarise.",
).compile()

app.invoke({"messages": [{"role":"user","content":"Plot US GDP since 2000."}]})


      

        
Deep dive — handoffs are just Command(goto=...)

        
A swarm handoff isn't magic: a handoff tool returns Command(goto="other_agent", update={...}),
          and the engine both jumps to that node and applies the state update in one move. You can write the
          same thing by hand without the prebuilt package — the supervisor/swarm libraries are just ergonomic wrappers
          over conditional edges and Command. Reach for the raw primitives the moment your routing needs
          something the wrapper doesn't express.

      


      

        
Two specialist agents mostly pass work straight to each other with no central decision-maker, and
          you want to minimise LLM calls. Which pattern fits?

        
    
          
  • Supervisor — always safest.
    • 
          
  • Swarm — peers hand off directly, avoiding an extra routing LLM hop each turn.
    • 
          
  • Neither; use a single agent with more tools.
    • 
        

        
Swarm's whole point is direct peer-to-peer handoffs (via Command), which
          removes the supervisor's per-turn LLM call — faster and cheaper when there's no genuine central routing
          decision to make. Supervisor wins when you do want that decision centralised and legible in traces.

      

    


    
    

      
Where people go wrong

      
Pitfall · overwriting state
        
Returning a whole list for a channel that has a reducer, expecting it to replace — or returning
          the full state instead of a partial update. Nodes return only the keys they changed; reducers decide the
          merge. Forgetting the reducer on messages silently wipes history.

      
Pitfall · unbounded loops
        
A cyclic graph with a flaky termination condition runs forever (and bills forever). Always bound it: a
          recursion_limit on invoke, a step/attempt counter in state, or a hard "give up and answer" edge.

      
Pitfall · HITL without persistence
        
Expecting interrupt() to work without a checkpointer + thread_id. No snapshot,
          nothing to resume. Same root cause behind "my agent forgot the conversation" — that's a missing/rotating
          thread id, not the model.

      
Pitfall · reaching for multi-agent too early
        
A supervisor over five agents when one agent with five tools would do. Every extra agent is extra latency,
          cost, and failure surface. Start single-agent; split only when a role genuinely needs its own prompt, tools,
          or state.

      
Pitfall · learning deprecated APIs
        
Copying create_react_agent tutorials into new code. It's deprecated in v1. Use
          create_agent; keep the old name only as reading knowledge.

1 · Self-correcting RAG Start here

        
Retrieve → grade the retrieved docs → if weak, rewrite the query or fall back to web search →
          generate → grade the answer → loop if it hallucinated. This is the "hello world" of useful
          LangGraph because it's inherently cyclic and conditional — a plain RAG chain can't self-correct.

2 · Coding agent with a reflection loop Core loop

        
Plan → generate code → run it against tests (a tool node) → if it fails, reflect on the traceback and
          retry, capped at N attempts. The failing test output feeds straight back into state — a concrete, verifiable
          reason to loop.

3 · "Agent inbox": human-approved actions HITL

        
An ops/support agent that drafts a consequential action (issue a refund, send an email, merge a PR), then
          interrupt()s for approval. Approve/edit/reject from a tiny UI; the run may sit paused for hours.
          The most employable pattern here — every real deployment needs a human gate.

4 · Multi-agent research assistant Supervisor

        
A supervisor routes to a search agent, a code/plotting agent, and a writer, then assembles a cited report.
          Make each specialist a subgraph so you can test them in isolation. Stream updates so the UI
          shows which agent is working.

5 · Parallel daily-briefing agent Fan-out/in
