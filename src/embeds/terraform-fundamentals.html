Infrastructure as Code · Terraform / HCL

Terraform, for someone who already knows Docker and Kubernetes

You've spent your career on the application side — FastAPI services, containers, clusters — and never had to provision the cloud underneath them. Good news: you already hold the hardest idea (declarative desired state; a machine reconciles it). Terraform is that idea applied to cloud APIs, with one twist Kubernetes doesn't have — a state file. Understanding why that file must exist is 70% of learning Terraform. The rest is syntax.

Full mode — deep-dive blocks visible

Why it exists

Before IaC, infrastructure was built by a human clicking through a console or running a shell script. Both fail in the same specific ways:

  • Not reproducible. The staging VPC and the prod VPC were built six months apart by two people. Nobody knows how they differ.
  • Not reviewable. There's no diff. A change to a security group has no pull request, no reviewer, no history.
  • Not idempotent. A bash script that runs aws ec2 run-instances creates a second instance when re-run. To make it safe you'd have to hand-write "does it exist? then what's different? then patch only that" — for every resource type.
  • Not disposable. Nobody dares delete anything, because nobody can rebuild it.

Terraform's pitch: describe the desired end state in files, commit those files, and let a tool compute the minimal set of API calls that moves reality to match. You get diffs, code review, git history, and a repeatable destroy.

The analogy that actually fits

A Kubernetes manifest is desired state; a controller loop watches and continuously reconciles. Terraform is desired state too — but there's no controller. You are the reconciliation loop. It runs when you type terraform apply and then stops caring. That single difference explains drift, state files, locking, and most of the operational pain.

Deep dive — why not just write the API calls yourself

You could. The reason nobody does is ordering and diffing. To create an EC2 instance you need a subnet; for the subnet you need a VPC; the security group needs the VPC id; the instance needs both subnet and SG ids. That's a dependency graph, and you'd have to topologically sort it by hand, then parallelise the independent branches, then handle partial failure, then figure out on the next run which of those 40 things already exist and which of their attributes drifted. Terraform's real product is that engine — a DAG builder plus a diff algorithm. The HCL syntax is just the input format.

The one idea: a diff engine over three worlds

Everything Terraform does is a comparison between three things. Hold this and the rest follows:

WorldLives inAnswers
Configyour .tf files, in git"What do I want?"
Stateterraform.tfstate (JSON), local or in S3/GCS"What do I manage, and what did it look like last time?"
Realitythe actual cloud, reachable via provider APIs"What's actually there right now?"

terraform plan = refresh state from reality, then diff config against state. terraform apply = execute that diff, then rewrite state.

Why state must exist

Kubernetes doesn't need a state file because the cluster is the state store — every object is stored under the name you gave it. AWS has no such thing. You call your instance aws_instance.web; AWS calls it i-0a1b2c3d. Nothing in AWS knows the string "web". State is the mapping from your names to their IDs — plus a cached snapshot of attributes so a diff is possible without re-reading the whole account. Delete the state file and Terraform doesn't "rediscover" your infra; it concludes you own nothing and offers to build it all again.

Watch the three worlds move

step 1 / 8

Watch specifically for steps 6 and 8: drift (reality changed behind your back) and amnesia (state was lost). They're the two failure modes that make state a real operational concern rather than an implementation detail.

Your colleague resizes an instance in the AWS console. Nobody touches the repo. You run terraform plan. What do you see?

  • No changes — the config never changed, so there's nothing to do.
  • A change that reverts the instance back to the size in your .tf file.
  • An error, because state no longer matches reality.
  • Terraform updates your .tf file to match the console.
Plan refreshes state from the API first, so state now says t3.large. It then diffs config (still t3.small) against that refreshed state, and plans to change it back. Config is the source of truth for intent; Terraform never writes your .tf files. This is exactly why "just make one quick fix in the console" is corrosive — the next apply silently undoes it.

The lifecycle: init → plan → apply → destroy

CommandWhat it actually does
terraform initReads required_providers, downloads provider plugins into .terraform/, writes/verifies .terraform.lock.hcl, initialises the backend, fetches modules. Run it after any provider/module/backend change.
terraform fmtCanonical formatting. Wire it into pre-commit; -check in CI.
terraform validateSyntax + internal consistency. No API calls, no credentials needed.
terraform planRefresh state from APIs, build the graph, compute the diff, print it. Read-only against your infra.
terraform applyPlan again, ask for confirmation, then execute the diff in dependency order and write state.
terraform destroyDiff config-is-empty against state. Walks the graph in reverse.

Reading a plan

# terraform plan
Terraform will perform the following actions:

  + aws_s3_bucket.logs        # create
  ~ aws_instance.web          # update in-place
      ~ instance_type: "t3.small" -> "t3.large"
  -/+ aws_instance.batch      # destroy and then create replacement
      ~ ami: "ami-111" -> "ami-222" # forces replacement
  - aws_eip.old               # destroy

Plan: 2 to add, 1 to change, 2 to destroy.
SymbolMeaningDanger
+createlow
~update in-place (a PATCH-style API call)low
-/+replace — destroy then create, because an attribute is immutable in the APIhigh — this is downtime
-destroyhigh
<=a data source will be readnone
Read every plan for -/+

Nothing in the syntax tells you which attributes force replacement — it's a property of the cloud API, encoded in the provider (an RDS identifier, an EC2 ami, an S3 bucket name). A one-character typo in a name can plan a destroy of your production database, and -auto-approve will happily do it. lifecycle { prevent_destroy = true } is your seatbelt.

Deep dive — plan is not a promise

A plan is computed at time T and applied at time T+n. In between, anything can change. Use a saved plan when it matters:

terraform plan -out=tfplan       # serialise the plan
terraform show -json tfplan | jq  # machine-readable — policy checks, cost estimation, CI gates
terraform apply tfplan            # apply exactly this, no re-prompt

Applying a saved plan still errors out if state has moved since (the plan records the state serial). That's the safety property CI pipelines rely on: plan on PR, human approves the artefact, apply that exact artefact on merge.

HCL: the whole syntax in one page

HCL is a declarative configuration language. It has exactly one shape — block_type "label" "label" { arguments } — and about eight block types you'll ever use. It is not a programming language: no loops in the imperative sense, no mutable variables, no if statements. Order in the file is irrelevant; the dependency graph decides execution order.

# ---------- 1. terraform: settings for Terraform itself ----------
terraform {
  required_version = ">= 1.10"

  required_providers {
    aws = {
      source  = "hashicorp/aws"   # registry.terraform.io/hashicorp/aws
      version = "~> 6.0"            # >= 6.0, < 7.0 — pessimistic constraint
    }
  }

  backend "s3" {                 # where state lives. NO variables allowed in here.
    bucket       = "rex-tfstate"
    key          = "prod/app/terraform.tfstate"
    region       = "ap-southeast-1"
    encrypt      = true
    use_lockfile = true          # S3-native locking (GA in 1.11; DynamoDB table no longer needed)
  }
}

# ---------- 2. provider: how to talk to a platform ----------
provider "aws" {
  region = var.region
  default_tags { tags = { managed_by = "terraform", repo = "rex/infra" } }
}

# ---------- 3. variable: typed inputs ----------
variable "env" {
  type        = string
  description = "Deployment environment"
  default     = "dev"
  validation {
    condition     = contains(["dev", "stg", "prod"], var.env)
    error_message = "env must be dev, stg or prod."
  }
}

variable "db_password" {
  type      = string
  sensitive = true    # redacts from CLI output. Does NOT encrypt it in state.
}

# ---------- 4. locals: named expressions (DRY, not inputs) ----------
locals {
  name_prefix = "${var.env}-app"
  is_prod     = var.env == "prod"
}

# ---------- 5. data: read something Terraform does NOT manage ----------
data "aws_ami" "ubuntu" {
  most_recent = true
  owners      = ["099720109477"]
  filter {
    name   = "name"
    values = ["ubuntu/images/hvm-ssd/ubuntu-jammy-22.04-amd64-server-*"]
  }
}

# ---------- 6. resource: something Terraform DOES manage ----------
resource "aws_instance" "web" {
  #        ^type          ^local name  --> address: aws_instance.web
  ami           = data.aws_ami.ubuntu.id     # <-- this reference creates a dependency edge
  instance_type = local.is_prod ? "m5.large" : "t3.micro"
  subnet_id     = aws_subnet.a.id

  tags = { Name = "${local.name_prefix}-web" }

  lifecycle {
    create_before_destroy = true
    prevent_destroy       = false
    ignore_changes        = [tags["LastScanned"]]   # stop fighting another system
  }
}

# ---------- 7. output: values out of this configuration ----------
output "web_ip" {
  value       = aws_instance.web.public_ip
  description = "Public IP of the web box"
}

# ---------- 8. module: reuse a folder of the above ----------
module "network" {
  source  = "terraform-aws-modules/vpc/aws"
  version = "~> 5.0"
  name    = local.name_prefix
  cidr    = "10.0.0.0/16"
}
resource vs data

resource = "I own this; create, update and delete it, and record it in state." data = "someone else owns this; just look it up every plan." Data sources are read-only and are how you reference things another team's Terraform (or a human) created.

Why can't the backend "s3" block use var.env in its key?

  • Because backends only accept string literals for security reasons.
  • Because Terraform must load state before it can evaluate anything — and variables are part of what it evaluates.
  • Because var.env isn't in scope inside terraform {} blocks.
  • It can; you just need terraform init -upgrade.
It's a chicken-and-egg: the backend has to be initialised before any evaluation happens, so nothing dynamic is available yet. The standard workaround is partial configuration — leave the values out and pass them at init: terraform init -backend-config=env/prod.tfbackend. (Notable divergence: OpenTofu 1.8 added early variable evaluation, which does allow variables in backend blocks — config using it won't run on Terraform.)

count vs for_each — the single most consequential choice

Both create multiple instances of one resource block. They differ in how instances are addressed in state, and that difference bites hard.

# count -> addressed by INDEX: aws_iam_user.u[0], u[1], u[2]
resource "aws_iam_user" "u" {
  count = length(var.usernames)
  name  = var.usernames[count.index]
}

# for_each -> addressed by KEY: aws_iam_user.u["alpha"], u["beta"]
resource "aws_iam_user" "u" {
  for_each = toset(var.usernames)
  name     = each.key          # each.key / each.value
}

Remove "beta" from the middle of the list. With count, every index after it shifts — Terraform sees u[1] change from beta to gamma and u[2] disappear, so it destroys and rebuilds resources that you never intended to touch. With for_each, only u["beta"] is destroyed. Run it below and watch the plans differ.

Rule of thumb
        
Use for_each for a collection of distinct named things (users, buckets, environments, containers).
          Use count only for genuinely interchangeable clones, or as an on/off switch:
          count = var.enabled ? 1 : 0. Switching an existing resource from count to
          for_each changes every address — use moved blocks or you'll rebuild the world.


      

        
You have count = length(var.subnet_cidrs) with 4 subnets, each holding a running database. You delete the 2nd CIDR from the list. What does plan show?

        
    
          
  • 1 to destroy — the second subnet.
    • 
          
  • Roughly 2 to replace and 1 to destroy — indices 1 and 2 get rewritten and index 3 vanishes.
    • 
          
  • No changes until you also update count manually.
    • 
          
  • An error about a shifted index.
    • 
        

        
Index-based addressing means removing a middle element shifts everything after it. [1] now
          holds what [2] held, [2] holds what [3] held, and [3] no longer exists.
          Since subnet CIDR is immutable, those "changes" are replacements — destroying live infrastructure. This is the classic
          3am incident, and it's why for_each is the default recommendation for anything named.

      

    


    
    

      
The dependency graph

      
You never declare ordering. Terraform infers it from references: writing subnet_id = aws_subnet.a.id
        creates an edge from aws_subnet.a to aws_instance.app. It then applies independent nodes in
        parallel (default -parallelism=10) and walks the graph in reverse on destroy.

State, properly

      
terraform.tfstate is JSON. Four things you must internalise:

      
    
        
  1. It's the source of truth for ownership. "Is this mine?" is answered by state, not by the cloud. Nothing else knows.
    2. 
        
  2. It contains secrets in plaintext. Every attribute of every resource is cached — RDS passwords, private keys, TLS certs.
          sensitive = true only redacts CLI output. Never commit state to git. Put it in an encrypted, versioned,
          access-controlled bucket.
    3. 
        
  3. It must be locked. Two concurrent applies = two writers = corrupted state.
    4. 
        
  4. It's the thing you'll have to surgically repair. Learn the commands before you need them at 3am.
    5. 
      


      
Remote backend — the current shape on AWS

      
terraform {
  backend "s3" {
    bucket       = "rex-tfstate"
    key          = "prod/app/terraform.tfstate"
    region       = "ap-southeast-1"
    encrypt      = true
    use_lockfile = true     # S3-native lock: writes a .tflock object using a conditional PUT
  }
}

      
Note for anyone reading older tutorials: state locking used to require a separate DynamoDB table
        (dynamodb_table). Native S3 locking arrived experimentally in Terraform 1.10 and went GA in 1.11; the
        DynamoDB arguments are deprecated and slated for removal. New backends shouldn't create a lock table at all.

}

      
These beat the imperative state mv/rm commands because they live in git, get code-reviewed, and run
        identically in CI. That's the whole point of IaC — apply it to your state operations too.


      

        
You run terraform state rm aws_instance.web. What happens to the EC2 instance?

        
    
          
  • It's terminated immediately.
    • 
          
  • It's terminated on the next apply.
    • 
          
  • Nothing — it keeps running, but Terraform now considers it none of its business.
    • 
          
  • It's tainted and will be replaced next apply.
    • 
        

        
state rm only edits the ledger. The instance keeps running and now nothing manages it —
          you've created an orphan that will quietly cost money forever. Worse: if the resource is still in your config,
          the next plan sees it missing from state and offers to create a second one. This is the intended
          tool for handing a resource to another config, but it's a footgun everywhere else.

      


      

        
Which of these is the strongest argument for splitting one big state file into several?

        
    
          
  • Terraform has a hard limit on state file size.
    • 
          
  • Blast radius and lock contention — one state means one lock, one plan over everything, and one mistake reaching everything.
    • 
          
  • Providers can only be configured once per state.
    • 
          
  • Modules require a state file each.
    • 
        

        
There's no size limit, and modules share their root's state. The real reasons are operational:
          every plan refreshes every resource (slow), one lock serialises the whole team, and a bad apply can touch
          anything in the file. The usual split is by rate of change and blast radius — network/IAM (rarely changes) separate
          from apps (changes hourly) — wired together with terraform_remote_state data sources or, better, by
          looking things up with normal data sources.

Modules

      
A module is just a folder with .tf files. That's the entire concept. The folder you run terraform apply
        in is the root module; anything it calls is a child module. Variables are the function parameters, outputs
        are the return values, and there's no state of its own — everything lands in the root's state under a prefixed address
        (module.network.aws_vpc.main).


      
# modules/site/variables.tf
variable "name" { type = string }
variable "port" { type = number }

# modules/site/main.tf
resource "docker_container" "this" {
  name  = var.name
  image = var.image_id
  ports { internal = 80, external = var.port }
}

# modules/site/outputs.tf
output "url" { value = "http://localhost:${var.port}" }

# root main.tf — calling it
module "site" {
  source   = "./modules/site"       # local path, git URL, or registry ref
  for_each = var.sites                # modules take for_each/count/depends_on too
  name     = "tf-${each.key}"
  port     = each.value.port
  image_id = docker_image.nginx.image_id
}

output "urls" { value = { for k, m in module.site : k => m.url } }

Don't over-modularise on day one
        
The common failure is wrapping every resource in a module "for reusability", producing a config where finding what
          actually gets created means opening six files. A module earns its existence when it's used twice (or when it
          enforces a real standard across teams). Start with a flat root module; extract later — moved blocks make
          that refactor safe, which is exactly what Lab 3 practises.


      

        
Where does the state for module.network.aws_vpc.main live?

        
    
          
  • In a state file inside modules/network/.
    • 
          
  • In the root module's single state file, under a prefixed address.
    • 
          
  • In whichever backend the child module declares.
    • 
          
  • Nowhere — child modules are stateless.
    • 
        

        
A child module has no state, no backend, and no provider configuration of its own — those are all
          root-level concerns. It's closer to a function call than to a microservice. Which is also why "just add a module" never
          reduces blast radius: the resources still land in the same state file, behind the same lock.

Real resources with the Docker provider

        
Same loop, but now Terraform manages actual running containers — you already have Docker on your Mac, so this is the

    external = each.value.port

  2. In the root, replace the docker_container.site resource with a module "site" call using the
            same for_each.
    3. 
          
  3. terraform init (new module ⇒ re-init) then plan. Read it carefully: it wants to destroy
            all your containers and create identical ones. Do not apply. Q: why? Nothing about the containers changed.
    4. 
          
  4. Add moved blocks and re-plan:
            
    moved {
  from = docker_container.site["alpha"]
  to   = module.site["alpha"].docker_container.this
}
moved {
  from = docker_container.site["beta"]
  to   = module.site["beta"].docker_container.this
}
    
          
    5. 
          
  5. Now plan says 0 to add, 0 to change, 0 to destroy — just address moves. Apply it. docker ps: same
            container IDs, never restarted. Q: what did moved actually modify — the containers, or the ledger?
    6. 
          
  6. Once applied, the moved blocks can be deleted in a later commit. Q: why keep them around for one release cycle?
    7. 
        

        
This lab is the single highest-leverage Terraform skill. Every real codebase gets refactored, and
          the difference between a senior and a junior Terraform user is whether that refactor causes an outage.

Adopt something that already exists (import)

        
The situation you'll actually walk into at work: infrastructure exists, nobody wrote it in code, and you're asked to
          bring it under management.

        
    
          
  1. Create a container outside Terraform: docker run -d --name legacy-box -p 8099:80 nginx:1.27-alpine
    2. 
          
  2. Get its id: docker inspect --format='{{.Id}}' legacy-box

    import {
  to = docker_container.legacy
  id = "<paste the full container id>"
}

            HCL for you by reading the live resource.
    5. 
          
  5. Clean up the generated file (it's verbose and includes computed junk), then terraform apply.
            Q: the apply reports "1 imported" — was any container created or restarted?
    6. 
          
  6. terraform plan again. Q: if it isn't clean, what does that tell you about the difference between
            what the API reports and what your HCL says?
    7. 
          
  7. Finish with removed { from = docker_container.legacy, lifecycle { destroy = false } } and apply.
            Q: is legacy-box still running? What just changed?

Where people go wrong


      
1 · State is a security asset
        
Plaintext secrets, in a file, that new engineers casually cat. Encrypt the bucket, restrict IAM on it,
          enable versioning (your undo button), add *.tfstate* to .gitignore on day one. Prefer
          ephemeral values and write-only arguments where the provider supports them, and generate secrets in a
          secrets manager rather than in Terraform.


      
2 · -auto-approve as a habit
        
Fine in a CI pipeline applying a plan a human already approved. Not fine in your terminal. The confirmation prompt is
          the last thing standing between a typo and a deleted database.


      
3 · Renaming a resource in code
        
aws_instance.web → aws_instance.frontend is not a rename to Terraform. It's a destroy and a
          create, because the address is the identity. Always pair a rename with a moved block.


      
4 · Not pinning, or not committing the lock file
        
version = "~> 6.0" in required_providers and .terraform.lock.hcl committed to git.
          Without both, CI can silently pull a new provider minor with different defaults and plan a change nobody wrote.


      
5 · Using Terraform as a config management tool
        
provisioner "remote-exec" is officially a last resort. Terraform provisions the box; Ansible/cloud-init/a
          container image configures what's on it. Provisioners run once at create time, aren't in the diff, and turn your
          declarative config into a fragile shell script.


      
6 · One state file for everything
        
It starts convenient and ends as a 20-minute plan, a permanently contended lock, and a blast radius covering prod.
          Split by rate-of-change and blast radius before it hurts, not after.


      

        
Deep dive — the misconceptions specifically worth naming

        
    
          
  • "Terraform reads my cloud account." No. It reads state, then refreshes only the resources listed there.
            Anything not in state is invisible to it.
    • 
          
  • "sensitive = true encrypts it." No. It redacts CLI/log output. State is still plaintext.
    • 
          
  • "Modules isolate blast radius." No. Same state, same lock, same apply.
    • 
          
  • "Plan means it will work." No. The provider can still get a 403, a quota error, or an eventual-consistency
            failure at apply time. Plan validates intent, not outcome.
    • 
          
  • "terraform destroy deletes everything I built." Only what's in this state file. Things you
            state rm'd, or that another config owns, survive — which is usually the surprise, not the relief.
    • 
        

      


      

        
CI runs terraform apply -auto-approve on merge to main. A PR renames aws_db_instance.main to aws_db_instance.primary, no other change. What ships?

        
    
          
  • Nothing — it's cosmetic, the address isn't part of the resource.
    • 
          
  • An in-place update to the database's identifier.
    • 
          
  • The production database is destroyed and a new empty one is created.
    • 
          
  • An error, because two addresses would collide.
    • 
        

        
The address is the identity in state. The old address vanishes from config ⇒ destroy; the new address
          has no state ⇒ create. Terraform is doing exactly what it was told. Three things would have stopped it: a
          moved block, lifecycle { prevent_destroy = true }, or a human reading the plan before apply.
          Note that all three are cheap, and that the reviewer saw a one-word diff.

      

    


    
    

      
When not to use it, and what else is out there

      
Terraform is a default, not a law. Where it's genuinely weak:

      

        
SituationWhy Terraform strugglesUsual answer

        
In-cluster Kubernetes workloadsNo continuous reconciliation; drift only surfaces when you run plan; the K8s provider fights the cluster's own controllersArgo CD / Flux (GitOps), or Crossplane if you want cloud resources reconciled by a K8s controller loop

        
OS / app configurationNot its model at allAnsible, cloud-init, a container image

        
Highly dynamic or short-lived resourcesState is a bottleneck; a lock per change doesn't scaleThe platform's own API / a controller

        
Complex conditional logicHCL deliberately isn't a programming language; you'll fight itPulumi or CDKTF (real languages — at the cost of a larger blast radius for bugs)

        
Single-cloud, deep AWS shopNothing wrong — but you're maintaining state yourselfAWS CDK / CloudFormation (AWS manages the state; you lose multi-cloud and the plan UX)

      

      
The genuine strength is the opposite case: heterogeneous providers (AWS + Cloudflare + Datadog + GitHub + Snowflake in
        one graph), an enormous provider ecosystem, and a plan output that a human can review. For your world specifically, the
        common split is Terraform for the cluster, node pools, IAM and networking; GitOps for everything inside the cluster.


      
Terraform or OpenTofu?

      
You'll hit this on day one, so here are the facts rather than a recommendation.

      
In August 2023 HashiCorp relicensed Terraform from MPL 2.0 to the Business Source License (BSL/BUSL), effective from
        v1.6. The community forked the last MPL version; the Linux Foundation adopted it as OpenTofu, which later entered
        the CNCF and stayed on MPL 2.0. IBM subsequently acquired HashiCorp, so Terraform is now an IBM product. Both are
        production-grade; tofu takes the same commands, reads the same state format, and uses the same providers.

      

        
AxisTerraformOpenTofu

        
LicenseBUSL 1.1 (not OSI-approved; converts to Apache 2.0 after 4 years)MPL 2.0 (OSI-approved)

        
GovernanceIBM / HashiCorpLinux Foundation, multi-vendor

        
Practical BUSL impactNone if you use it to manage your own infra. Real if you build a product that competes with HCP Terraform.None

        
Exclusive featuresStacks (.tfstack.hcl), deep HCP Terraform + Sentinel integrationClient-side state encryption, for_each on providers, early variable evaluation, -exclude

        
EcosystemLargest; every tutorial and module assumes itSame providers; supported by Spacelift, Scalr, env0, Atlantis

      

      
Honest reading of the evidence: for learning, it doesn't matter — everything in this module works in both.
        For a team, the deciding axes are your legal/procurement posture on BUSL, whether you depend on HCP Terraform features,
        and how much you value vendor-neutral governance. Be aware that much of the published comparison is written by vendors
        with a commercial stake (Scalr and Spacelift back OpenTofu; HashiCorp is now IBM), so weigh sources accordingly. The
        divergence is real and growing, but the overlap is still large enough that switching is currently cheap.

    


    
    

      
Check yourself


      

        
A resource is in your config and running in the cloud, but its state entry was lost. What does terraform apply do?

        
    
          
  • Detects the existing resource and adopts it automatically.
    • 
          
  • Tries to create a second one — and probably fails on a name collision, or succeeds and gives you a duplicate.
    • 
          
  • Errors out saying state is inconsistent.
    • 
          
  • Nothing; reality already matches config.
    • 
        

        
Terraform has no discovery mechanism. State is the only thing that says "this is mine, here's its
          real id". Without it, the resource is invisible and the config looks unfulfilled. The fix is an import
          block, which is precisely why Lab 4 exists — and why remote state gets versioning turned on.

      


      

        
Which change is safe to apply without reading the plan?

        
    
          
  • Adding a tag to an existing instance.
    • 
          
  • Bumping a provider version.
    • 
          
  • Adding a new S3 bucket.
    • 
          
  • None of them.
    • 
        

        
Not a trick question — a discipline one. Adding a tag can force replacement on some resource types.
          A provider bump can change defaults and produce diffs you never wrote. A "new" bucket whose name collides fails the
          apply halfway, leaving a partial change. The plan takes ten seconds to read and it is the entire safety model.

      


      

        
Terraform applies aws_vpc, aws_subnet (refs the VPC) and aws_s3_bucket (refs nothing). What runs in parallel?

        
    
          
  • Nothing — Terraform applies strictly in file order.
    • 
          
  • The VPC and the bucket start together; the subnet waits for the VPC.
    • 
          
  • All three, since only depends_on creates ordering.
    • 
          
  • Nothing, unless you pass -parallelism.
    • 
        

        
Order in the file is irrelevant — the graph decides. The reference vpc_id = aws_vpc.main.id
          is what creates the edge, and it's implicit. The bucket shares no edges, so it's in the first wave alongside the VPC.
          Parallelism defaults to 10 and only caps the width of each wave.
